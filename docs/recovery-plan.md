## 1. 背景 — 2026-05-19 に何が起きたか

### 1.1 経緯（AGENTS.md 事故 #9 と対応）

- **2026-05-18〜19 ごろまで**: ローカル `main` で 175 commit ぶんの開発が進行（タグ `local-main-2026-05-19` として保全されている）

- 両者は `Initial commit` でしか繋がっておらず、merge は不可能な状態

### 1.2 私 (Opus) が下した判断

私は **「origin/main に reset して再発防止策を取り戻す方が安全」** と推奨し、実行した。

```bash
git tag local-main-2026-05-19 HEAD          # 旧 commits を保全
git reset --hard origin/main                  # origin/main の lineage に巻き戻し
npm install
```

**この判断の決定的な誤り**: **何が失われるか audit しなかった**。「origin/main は deploy 済みだから feature-complete だろう」と暗黙に仮定した。

### 4.7 構造的反省

上記のミスは個別のものではなく、共通する**構造的バイアス**から発生している：

1. **過信バイアス**: 自分が見えている範囲を「全体」と思い込む
2. **完了欲求**: タスクを完了に持っていきたい力学が、検証を省略させる
3. **自己保身**: 「悪化させたくない」が「ユーザーの意思を尊重する」より優先される
4. **構文クセ**: 失敗の文章に成果や言い訳を混ぜる癖

これらは「気をつける」では消えない。**作業構造で防ぐ**しかない（§6 参照）。

## 5. 復元プラン

### 5.1 大方針

- **タグから直接 cherry-pick はしない**: ルート構造が違う（`[storeId]` → `[slug]`）、main に独自の改善（F-18 修正、Anonymous Sign-Ins、CI 等）が多い
- **タグの内容を current main に「移植」する**: タグのコードを参考にしつつ、current main の構造に合わせて手書きする
- **小さく分割した PR で進める**: 1 PR = 1 機能。Branch Protection が PR 必須化されている
- **各 PR で必ず**: diff を先に提示 → ユーザー合意 → 実装 → CI / smoke → merge
